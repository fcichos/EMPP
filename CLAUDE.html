    Project Overview

    This is EMPP (Einf√ºhrung in die Modellierung Physikalischer Prozesse / Introduction to Modeling Physical Processes), a Python-based computational physics course taught in German at a university level. The course website is built with Quarto and contains lecture materials (.qmd files) and seminar exercises covering topics from basic Python to advanced physics simulations.

    Repository: https://github.com/fcichos/EMPP Website: https://fcichos.github.io/EMPP/ Primary Language: Course is taught in German, but code/documentation may be in English.

    Building and Development

    Build the Website

    # Activate UV environment
default

# Render the website
quarto render

    This renders all .qmd files to HTML and outputs to the docs/ directory.

    Note: The _quarto.yml does NOT have an explicit render: section. Quarto auto-discovers all .qmd files in the project. Do not add a render: list unless you need to exclude specific files.

    Launches a local server with live reload.

    Render Specific Files

    quarto render lectures/lecture01/jupyter-notebooks.qmd

    Course Structure and Pedagogy

    Key Pedagogical Principles

    1. Spiral Learning: Topics like Brownian motion are taught twice:
      • Week 2-3 (lecture02/brownian-motion-simple.qmd): Simple functional approach, focus on physics
      • Week 6-7 (lecture05/brownian-motion-advanced.qmd): Object-oriented refactor, focus on code organization
    2. Progressive Difficulty: Week 6 is carefully ordered:
      • First: Planetary motion (stable, intuitive, single force)
      • Second: Spring pendulum (chaotic, complex, coupled forces)
      • This creates confidence before complexity
    3. Physics First, Code Later: Students learn physics with simple code, then refactor with better practices when they understand why organization matters.

    Content Flow

    Week 1  ‚Üí Setup: Jupyter notebooks, first plots
Week 2-3 ‚Üí Python basics: variables, datatypes, functions
Week 4  ‚Üí NumPy arrays for physics

Week 6  ‚Üí APPLY Week 5 tools: Planetary motion + Spring pendulum

Week 8  ‚Üí OOP applications: Advanced Brownian motion, I/O

Week 11-12 ‚Üí Review and repetition
Week 13-14 ‚Üí PDEs: Diffusion, quantum mechanics

    Working with Quarto Documents

    File Format

    • Lectures are .qmd (Quarto markdown) files combining markdown, LaTeX, and Python code
    • Code blocks use {python} syntax
    • Support for both HTML and PDF rendering

    Common Patterns

    Python code blocks:

    #| echo: true
#| eval: true

    Slides format (for some lectures):

    ---

    Unused/Archive Files

    • Files in _unused/ or lectures/lectureXX/_unused/ directories are archived previous versions
    • These contain old approaches that were pedagogically replaced
    • Do NOT delete without understanding the restructuring rationale documented in PEDAGOGICAL_NOTES.md

    Key Technical Decisions

    Why Lecture Numbers Don‚Äôt Match Week Numbers

    The lecture directories (lecture01-lecture15) don‚Äôt directly map to weeks because: - Content was reorganized for pedagogical flow - Some lectures span multiple weeks - Review weeks reuse earlier lecture materials - Always refer to _quarto.yml for the actual week-to-content mapping

    Why Two Brownian Motion Lectures

    See lectures/PEDAGOGICAL_NOTES.md for full rationale: - Teaching OOP in Week 2-3 creates cognitive overload - Students learn physics with functions first - Later revisit with classes when they understand WHY organization helps - Direct comparison shows value of OOP

    Why Planetary Motion Before Spring Pendulum

    See LECTURE_ORDER_RATIONALE.md for full rationale: - Planetary motion: stable, intuitive, single force (confidence builder) - Spring pendulum: chaotic, complex, coupled forces (interesting challenge) - Natural progression from simple ‚Üí complex

    Common Tasks

    Adding New Lecture Content

    1. Create .qmd file in appropriate lectures/lectureXX/ directory
    2. Add entry to _quarto.yml in correct week section
    3. Test render: quarto render lectures/lectureXX/new-file.qmd
    4. Full site render: quarto render

    Modifying Course Structure

    1. Always read pedagogical docs first: PEDAGOGICAL_NOTES.md, LECTURE_ORDER_RATIONALE.md
    2. Update _quarto.yml to reflect changes
    3. Consider downstream effects on dependent lectures
    4. Update any cross-references in other files

    Testing Changes

    # Render and preview locally
quarto preview

# Check for broken links/references
quarto check

# Render full site
quarto render

    Publishing to GitHub Pages

    Standard Quarto Workflow:

    # Activate UV environment
default

# Publish to GitHub Pages
quarto publish gh-pages

    This command: 1. Renders the site 2. Commits to gh-pages branch 3. Pushes to GitHub 4. GitHub Pages automatically deploys

    Alternative: Manual Publishing

    If quarto publish gh-pages has issues, use manual git workflow:

    # 1. Build
default && quarto render

# 2. Commit changes
git add docs/
git commit -m "Update website"

# 3. Push to GitHub
git push origin main

    GitHub Pages is configured to serve from the docs/ directory on the main branch.
