Brownian Motion with Object-Oriented Programming

Remember simulating Brownian motion back in Week 2 using functions and lists? That worked, but as simulations get more complex (different particle sizes, tracking many properties, adding interactions), managing separate lists for x-positions, y-positions, radii, and other properties becomes messy.

Now that you‚Äôve learned about classes and object-oriented programming, let‚Äôs revisit Brownian motion with a better approach! This time, each particle will be an object that ‚Äúknows‚Äù its own properties (size, position, diffusion coefficient) and behaviors (how to update, how to calculate its trajectory). You‚Äôll see how OOP makes the code cleaner, easier to extend, and much more maintainable.

Applying What You‚Äôve Learned

This lecture puts into practice everything from the previous lecture on classes and objects:

  • Instance variables: Each particle has its own radius, position, and diffusion coefficient
  • Class variables: We‚Äôll track the total number of particles across all instances
  • Methods: Particles will have behaviors like update() and sim_trajectory()
  • Special methods: We‚Äôll use __init__ and __str__ for initialization and display
  • Inheritance: Later, we‚Äôll extend the base class to create specialized particle types

Watch for these concepts as they appear in the code!

Comparing Approaches: Procedural vs. Object-Oriented

Week 2 approach (Procedural):

# Managing particles with separate lists
all_x = []  # List of x trajectories
all_y = []  # List of y trajectories
radii = []  # List of particle sizes
# ... nightmare when adding more properties!

# Simulate each particle
for i in range(n_particles):
    x, y = simulate_trajectory(x0, y0, D, dt, N)
    all_x.append(x)
    all_y.append(y)

Week 5 approach (Object-Oriented):

# Each particle is an object with its own properties
particles = []
for i in range(n_particles):
    p = Colloid(R=1e-6)  # Particle knows its size, position, D
    p.sim_trajectory(N, dt)  # Particle updates itself!
    particles.append(p)

Benefits of OOP:

  • Each particle manages its own data (no juggling multiple lists!)
  • Easy to add properties (just add to __init__)
  • Particles can have different sizes with different diffusion coefficients
  • Methods are attached to the objects they operate on
  • Much easier to extend (add charged particles, magnetic particles, etc.)

Quick Physics Review

You already learned the physics of Brownian motion in Week 2, so here‚Äôs a quick refresher:

Key Physics Concepts:

  • Particles undergo random motion due to molecular collisions
  • Each step is random with size \(\sigma = \sqrt{2D\Delta t}\)
  • Mean squared displacement: \(\langle r^2 \rangle = 4Dt\) (in 2D)
  • Diffusion coefficient: \(D = k_BT/(6\pi\eta R)\) depends on particle size \(R\)

What‚Äôs Different This Time:

In Week 2, you simulated particles with the same size. Now with OOP, we can easily simulate particles with different sizes, each with its own diffusion coefficient! This would be much messier with the procedural approach.

Advanced Mathematical Details

The Brownian motion of a colloidal particle results from collisions with surrounding solvent molecules. These collisions lead to a probability distribution described by:

\[ p(x,\Delta t)=\frac{1}{\sqrt{4\pi D \Delta t}}e^{-\frac{x^2}{4D \Delta t}} \]

where:

  • \(D\) is the diffusion coefficient
  • \(\Delta t\) is the time step
  • The variance is \(\sigma^2=2D \Delta t\)

This distribution emerges from the central limit theorem, as shown by Lindenberg and L√©vy, when considering many infinitesimally small random steps.

The evolution of the probability density function \(p(x,t)\) is governed by the diffusion equation:

\[ \frac{\partial p}{\partial t}=D\frac{\partial^2 p}{\partial x^2} \]

This partial differential equation, also known as Fick‚Äôs second law, describes how the concentration of particles evolves over time due to diffusive processes. The Gaussian distribution above is the fundamental solution (Green‚Äôs function) of this diffusion equation, representing how an initially localized distribution spreads out over time.

The connection between the microscopic random motion and the macroscopic diffusion equation was first established by Einstein in his 1905 paper on Brownian motion, providing one of the earliest quantitative links between statistical mechanics and thermodynamics.

Why Use a Class for Brownian Motion?

Think about what a real colloidal particle does:

  • It has a specific size (radius)
  • It has a current position in space
  • It remembers where it‚Äôs been (trajectory)
  • It knows how fast it diffuses (based on its size)
  • It can update its position when time passes

In Week 2, we used separate variables and functions for all this. But doesn‚Äôt it make more sense to bundle all of a particle‚Äôs data and behaviors into one object?

Object-Oriented Approach:

particle = Colloid(R=1e-6)           # Create particle with size
particle.sim_trajectory(N=200, dt=0.05)  # Particle updates itself
print(particle)                       # Particle describes itself
trajectory = particle.get_trajectory() # Get its history

Compare this to managing separate x_list, y_list, radii_list, and writing functions that need all these lists as parameters!

Key advantages for our simulation:

  1. Each particle is self-contained - no juggling multiple lists
  2. Easy to create particles with different sizes - each calculates its own D
  3. Natural syntax - particle.update() is intuitive
  4. Easy to extend - want charged particles? Just inherit from Colloid!

Class Design

Let‚Äôs design a Python class to simulate colloidal particles undergoing Brownian motion. This object-oriented approach will help us manage multiple particles with different properties and behaviors.

Class-Level Properties

The Colloid class will maintain information shared by all particles:

  1. A counter for the total number of particles
  2. The physical constant \(k_B T/(6\pi\eta) = 2.2√ó10^{-19}\) (combining temperature and fluid properties)

Class Methods

The class will provide these shared functions:

  1. how_many(): Reports the total number of particles
  2. __str__: Creates a readable description of a particle‚Äôs properties

Instance Properties

Each individual particle object will have:

  1. Radius (R)
  2. Position history (x and y coordinates)
  3. Unique identifier (index)
  4. Diffusion coefficient (\(D = k_B T/(6\pi\eta R)\))

Instance Methods

Each particle will be able to:

  1. sim_trajectory(): Generate a complete motion path
  2. update(dt): Calculate one step of Brownian motion
  3. get_trajectory(): Return its movement history
  4. get_D(): Provide its diffusion coefficient
OOP in Action

Notice several object-oriented features here:

  • Class variables: number and f are shared by all particles
  • Instance variables: Each particle has its own R, x, y, and D
  • Methods calling methods: sim_trajectory() calls update() on the same object
  • @classmethod decorator: how_many() operates on the class itself, not instances
  • __str__ method: Defines how particles are displayed when printed

Characterizing the Brownian motion

Now that we have trajectories, let‚Äôs analyze the motion. You learned these analysis techniques in Week 2, but notice how much cleaner the code is with OOP - each particle manages its own trajectory!

Review: Why Speed Doesn‚Äôt Work for Brownian Motion

As you saw in Week 2, calculating speed from Brownian trajectories gives unphysical results (infinite speed at short times). This is because Brownian motion is fundamentally a diffusive process, not ballistic motion. The correct measure is the mean squared displacement (MSD).

Calculate the particle mean squared displacement

The MSD is the right way to characterize diffusive motion. Recall from Week 2:

\[\begin{equation} \langle \Delta r^{2}(t)\rangle=2 d D t \end{equation}\]

where \(d=2\) for 2D motion. Let‚Äôs verify our simulation matches theory:

OOP Makes This Cleaner!

Notice how easy it is to get each particle‚Äôs trajectory:

t = p_list[j].get_trajectory()  # Each particle returns its own data!

In Week 2‚Äôs procedural approach, you had to manage separate all_x[j] and all_y[j] lists. The OOP approach is much cleaner and less error-prone.

Key Observations: - Individual particle MSDs fluctuate around the theoretical prediction - On average, MSD follows \(\langle r^2 \rangle = 4Dt\) - Fluctuations increase at long times (fewer data points) - For accurate D measurements, use many particles or early time points

For detailed statistical analysis, refer back to Week 2 lecture where you explored these concepts thoroughly.

Summary: What We‚Äôve Learned

In this lecture, we‚Äôve applied object-oriented programming principles to simulate Brownian motion of colloidal particles. Here‚Äôs what we covered:

Physics Concepts:

  • Brownian motion arises from molecular collisions with suspended particles
  • Motion is characterized by random steps following a normal distribution
  • The mean squared displacement (MSD) grows linearly with time: \(\langle \Delta r^2 \rangle = 2dDt\)
  • Speed is not a good measure for Brownian motion due to the random nature of the process

OOP Implementation:

  • Created a Colloid class with both instance variables (R, x, y, D) and class variables (number, f)
  • Used the constructor __init__ to initialize particles with specific properties
  • Implemented methods for simulation (update, sim_trajectory) and data access (get_trajectory, get_D)
  • Used special methods like __str__ for readable output
  • Applied inheritance to create ChargedColloid with additional properties and behaviors
  • Used class methods (@classmethod) to track global particle statistics

Computational Insights:

  • OOP makes managing multiple particles with different properties straightforward
  • Each particle encapsulates its own state and behavior
  • Inheritance allows us to extend functionality without modifying existing code
  • Statistical analysis requires careful consideration of finite sampling effects

The object-oriented approach makes the code more organized, maintainable, and extensible compared to procedural programming. When you need to add new particle types or behaviors, you can simply create new classes through inheritance rather than modifying existing code.
